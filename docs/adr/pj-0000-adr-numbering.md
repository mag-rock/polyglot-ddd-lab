# ADR pj-0000: ADR ナンバリング/プレフィックス方針

- 日付: 2025-11-26
- ステータス: 提案
- 適用範囲: プロジェクト全体
- 決定者: プロジェクトチーム（学習者承認前提）

## 背景
- プロジェクト共通の意思決定と、言語固有の意思決定を区別したい。
- マルチランゲージ（Kotlin / Go / Rust / TypeScript）で同一ドメインを扱うため、ADRの整理・検索性を担保する必要がある。

## 決定
- ADR ID はプレフィックスでスコープを区別し、ゼロパディングの連番を付与する。
  - 共通: `pj-000x`（プロジェクト全体に適用）
  - Kotlin: `kt-000x`
  - Go: `go-000x`
  - Rust: `rs-000x`
  - TypeScript: `ts-000x`
- 1 ADR = 1 ファイル。`docs/adr/<prefix>-<number>-<slug>.md` とし、ナンバリングの重複を避ける。
- 既存 ADR も順次この命名に合わせる（後続作業でリネーム）。

## 選定理由
- プレフィックスでスコープが一目で分かり、検索/参照性が向上する。
- マルチランゲージでの差分管理を容易にし、共通決定と個別決定の混在を防ぐ。
- ゼロパディング連番でソート順を安定させ、履歴を追いやすくする。

## 運用ルール
- 共通決定は `pj-`、言語固有は該当言語プレフィックスを使用する。
- 先着順で番号を確保し、PR/コミット単位で重複を避ける（必要ならプレフィックス別に番号予約可）。
- ステータスは `提案/承認/却下/廃止` を明記する。

## 影響範囲
- 既存 ADR ファイルの命名規則（後続でリネームが必要）。
- ADR 参照リンクやドキュメント内の参照はリネーム後に更新が必要。
